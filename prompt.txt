## Field Validation Rules & Constraints

### String Fields (NVARCHAR)
- Max length enforced by DB
- No leading/trailing spaces recommended
- Case-sensitive in some databases
- Special characters allowed unless validated at app level

### Numeric Fields
- **DECIMAL(15,2)**: Monetary values
  - Range: -999,999,999,999.99 to +999,999,999.99
  - Used for: Amount, POValue, BudgetValue
- **DECIMAL(13,3)**: Quantities
  - Range: -9,999,999,999.999 to +9,999,999.999
  - Used for: OrderQuantity, DispatchQuantity, Accepted/Rejected Qty
- **DECIMAL(3,1)**: Ratings
  - Range: -99.9 to +9.9
  - Typically 0.0 to 9.9 (vendor rating)
- **INTEGER**: Whole numbers, percentages
  - Range: -2,147,483,648 to 2,147,483,647
  - Used for: Progress %, Attempt counts
### Date/Timestamp Fields
- **DATE**: YYYY-MM-DD format (e.g., 2024-01-15)
- **TIMESTAMP**: YYYY-MM-DD HH:MM:SS.sssssss (27 chars, 7 decimals for microseconds)
- All dates in UTC recommended
- CREATEDAT/MODIFIEDAT auto-filled by system
### Boolean Fields
- Only in USERS table (ISPORTALUSER)
- Values: true/false or 1/0
- Never NULL if properly constrained

### Status Enumerations
Common status values (must be one of these):
**POStatus**: Pending | Confirmed | In Progress | Completed | Cancelled
**InspectionStatus**: Pending | In Progress | Completed | On Hold | Rejected
**DispatchStatus**: Pending | In Transit | Delivered | Returned
**PaymentStatus**: Pending | Due | Paid | Overdue | Cancelled
**QueryStatus**: Open | In Progress | Resolved | Closed | Reopened
**AssignmentStatus**: Assigned | In Progress | Completed | Cancelled

## Data Integrity Rules

1. **Foreign Keys Should Always Exist**

   - VENDORCODE in PO must exist in VENDORS
   - PROJECTID in PO must exist in PROJECTS
   - MATERIALCODE in PO must exist in MATERIALS
   - USERID references must exist in USERS
2. **Quantity Validations**
   - OpenQuantity ≤ OrderQuantity (always)
   - AcceptedQuantity + RejectedQuantity + HoldQuantity ≤ OfferedQuantity
   - DispatchQuantity ≤ OrderQuantity
3. **Date Validations**
   - PODate ≤ DeliveryDate
   - DueDate ≥ PaymentDate
   - ResolutionDate ≥ QueryDate (for closed queries)
4. **Amount Validations**
   - Amount > 0 (for payments)
   - POValue > 0
   - BudgetValue ≥ 0
5. **Status Flow**
   - Once status changes to "Completed" or "Cancelled", typically immutable
   - Status transitions should follow business rules

## Business Rules

**Happy Path - PO Lifecycle**:
1. Create PO (POStatus=Pending)
2. Confirm PO (POStatus=Confirmed)
3. Create Inspection (InspectionStatus=Pending)
4. Assign Inspectors (AssignmentStatus=Assigned)
5. Record Inspection Results (InspectionStatus=Completed)
6. Create Dispatch (DispatchStatus=Pending → In Transit → Delivered)
7. Create Payment (PaymentStatus=Pending → Paid)
8. Update ManufacturingProgress to 100%

**Issue Path - Query Management**:
1. Identify issue → Raise QueryList (Status=Open)
2. Route to stakeholder (QueryPendingWith=User)
3. Add QueryItems for specific line items

### Critical Validation Rules

✓ **Quantity Check**: OpenQuantity ≤ OrderQuantity (always)

✓ **Date Logic**: PODate ≤ DeliveryDate, DueDate ≥ PaymentDate

✓ **Amount Check**: All amounts must be positive

✓ **Status Transitions**:
  - Once "Completed" or "Cancelled" → immutable
  - Query must be "Open" before moving to "In Progress"
  - Inspect before Dispatch
